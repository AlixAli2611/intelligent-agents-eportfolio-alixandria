Testing and Remediation

Section focus: This page documents testing evidence, debugging, and remediation activities from the ResearchMate implementation.

Testing Evidence

The final codebase passed 24 pytest tests. These covered planning, processing, ranking, evidence assessment, output storage, SQLite persistence, validation, and fallback logic. Testing was important because agent-oriented development often becomes clearer as components are implemented and connected, rather than only at the design stage (He, Treude and Lo, 2024).

Testing Covered

  • Planning agent behaviour
  • Processing agent output
  • Ranking and recommendation logic
  • Evidence assessment behaviour
  • Markdown and JSON output generation
  • SQLite database persistence
  • LLM JSON validation and fallback logic

Remediation Activities

The remediation notes document issues encountered during development, including Groq JSON parsing, LLM output variability, Semantic Scholar rate limits, and accidental duplicate test discovery caused by a nested clone.

  • Ranking tests were refined to check meaningful behaviour rather than exact LLM wording.
  • Groq JSON parsing was improved to handle markdown fences or extra text.
  • Retrieval behaviour was improved to handle Semantic Scholar rate limits.
  • An accidental nested clone causing duplicate test discovery was removed.
  • SQLite persistence was added and tested.

Why Testing Mattered

Testing helped me see that an agent-based system needs repeatable behaviours, not just a successful demonstration once. Because LLM outputs can vary, the tests could not always rely on exact wording. Instead, they needed to check structure, behaviour, and meaning. This was an important learning point because intelligent agent systems often combine deterministic code with probabilistic model outputs.

Testing gave me confidence that ResearchMate worked as a structured prototype rather than only as a one-off terminal demonstration.
